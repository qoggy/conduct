# conduct 语言与国际化

本规格定义 conduct 的语言来源、文案分类、共享错误边界和持久化语义，覆盖 CLI、Web UI、HTTP API 与 `run-summary.md`。

## 核心决策

- conduct 生成的底层技术诊断、日志和浏览器 `console` 诊断固定使用英文。
- CLI 和 Web UI 的所有人类可读产品文案均支持中英文，并使用同一套语言解析规则。
- 语言优先级固定为 `~/.conduct/settings.json` 的 `language` > `LC_ALL` > `LC_MESSAGES` > `LANG` > 英文。
- Web UI 的设置页提供语言与主题选项；页面切换会修改全局 `settings.json`，语言设置同时影响之后启动的 CLI。
- 用户输入、AI 产物、原始引擎输出和第三方原文不由 conduct 翻译。

## 语言解析

每个 conduct 进程在启动时按以下顺序解析语言：

1. 读取 `~/.conduct/settings.json`；若 `language` 存在，使用该值。
2. 若 `language` 缺失，按 `LC_ALL` > `LC_MESSAGES` > `LANG` 取第一个非空值。
3. 若仍未得到可识别的语言，使用英文。

不增加应用专属语言环境变量、`--lang` 或另一份语言配置。

### settings.json

`settings.json` 是 conduct 全局设置容器，未来可增加其它属性。当前 schema：

```json
{
  "language": "zh-CN",
  "theme": "dark"
}
```

| 字段 | 类型 | 必填 | 允许值 | 含义 |
| --- | --- | --- | --- | --- |
| `language` | string | 否 | `en` / `zh-CN` | 显式覆盖环境变量；缺失即“跟随环境” |
| `theme` | string | 否 | `light` / `dark` | Web UI 显式主题；缺失即“跟随系统” |

文件不存在或 `language` 缺失都是正常状态，继续读环境变量。以下情况必须用固定英文技术诊断报错并停止当前命令，不得静默忽略或降级：

- 文件存在但无法读取。
- JSON 损坏或顶层不是对象。
- `language` / `theme` 不是字符串或不在各自允许值内。

例如：`failed to parse ~/.conduct/settings.json: ...` 或 `unsupported language "fr"`。由于语言本身未能成功解析，这类错误不可本地化。

未知的其它属性不报错。更新已支持的设置时必须先读取原对象、只修改/删除请求中出现的字段、保留其它属性，再通过同目录临时文件 + 原子替换写回。

写入显式设置时，若 `~/.conduct/` 或文件不存在则创建；切换为跟随项时，文件不存在即视为成功且不创建空文件，文件存在则仅删除对应属性。文件还含其它属性时不得删除整个文件。

### 环境变量

只有在 `settings.json` 没有 `language` 时才读环境变量：

- `zh` 及带地域、文字、编码后缀的常见中文 locale 归一为 `zh-CN`。
- `C`、`POSIX`、未设置或其它无法识别的值归一为 `en`。
- 高优先级值无法识别时直接使用 `en`，不继续读低优先级变量。

匹配不区分大小写。移除编码后缀（如 `.UTF-8`）和 modifier（如 `@variant`）后，以 `_` 或 `-` 分隔 locale；首段为 `zh` 即中文，首段为 `en`、`C`、`POSIX` 或其它值均为英文。只读取这三个变量，不读取 GNU gettext 的 `LANGUAGE`，也不支持 `CONDUCT_LANG`。

## 文案分类

| 类别 | 是否本地化 | 适用内容 |
| --- | --- | --- |
| 底层技术诊断 | 否，固定英文 | IO/路径/JSON/存储损坏/进程/信号/引擎适配器/内部不变量错误 |
| 日志与 console 诊断 | 否，固定英文 | 服务端日志、警告日志、浏览器 `console.error/warn` |
| CLI 产品文案 | 是 | help、用法/领域错误、校验、确认、成功/警告/空态、人读列表/详情/进度 |
| Web UI 产品文案 | 是 | 导航、标题、按钮、字段、表头、弹窗、空态、页面领域错误和前端校验 |
| 机器协议 | 否，值稳定 | 命令/flag、JSON 字段、状态枚举、engine/id/错误码、文件名与 schema |
| 外部/用户内容 | 否，原样保留 | 用户输入、节点名、prompt、AI 产物、原始引擎/第三方错误 |

“固定英文”只约束 conduct 自己生成的技术诊断模板。经 `%w` / `%v` 包装的操作系统、引擎或第三方原始文本必须原样保留，conduct 不保证原文本身一定是英文。UI 可在技术详情前显示已本地化的产品层概要，但技术详情本身不翻译。

### 边界判断

文案属于哪一类由语义决定，不由输出通道决定：写到 `stderr` 的领域错误仍是产品文案；显示在浏览器里的底层技术详情仍是固定英文。

- 描述用户正在操作的 conduct 领域对象、合法用法或可采取动作，属于产品文案。
- 描述 conduct 的实现细节、文件/JSON/进程/协议解析、引擎适配或内部不变量，属于技术诊断。
- 内容由用户、AI 引擎、操作系统或第三方产生，属于外部原文；conduct 只翻译自己添加的产品层外壳。

| 例子 | 分类 | 规则 |
| --- | --- | --- |
| `workflow <name> does not exist` / `工作流 <name> 不存在` | CLI/UI 领域文案 | 随语言切换 |
| help 中的说明、示例注释与自然语言占位符 | CLI 产品文案 | 忠实维护中英文；机器语法不变 |
| 页面按钮、表头、弹窗、空态和前端字段校验 | UI 产品文案 | 随语言切换 |
| `failed to resolve --cwd path: ...` | 技术诊断 | 固定英文 |
| `claude error: ...` | 引擎适配器诊断 | conduct 前缀固定英文，原始引擎文本原样保留 |
| `qodercli returned no specific error information` | 引擎适配器诊断 | 固定英文 |
| `agy passes the prompt via a command-line argument, and the prompt is too long (...)` | 引擎适配器诊断 | 固定英文 |
| `codex returned unexpected JSON: failed to parse line ...` | 协议解析诊断 | 固定英文 |

## CLI 产品文案

下列人类可读内容必须同时维护中英文，并忠实对应：

- help 与 help topic。
- 未知命令/flag、缺少/非法参数等用法错误。
- 资源不存在/已存在、状态不允许、校验失败等领域文案。
- 确认、成功、警告、空态、更新提示、workflow 进度和 run 人读输出。
- 新生成的 `run-summary.md` 中由 conduct 提供的标题、表头、单位和提示。

`--json` 的字段名、枚举值和结构不本地化。命令失败时，领域文案本地化，技术诊断固定英文。

## Web UI 产品文案与设置

顶栏右侧提供设置入口，进入 `#/settings`。设置页的语言选项为：

| 持久化值 | 中文界面标签 | English UI label | 行为 |
| --- | --- | --- | --- |
| 属性缺失 | 跟随环境 | Follow environment | 删除 `settings.json` 的 `language`，当前 UI 使用启动 `conduct ui` 时的环境变量解析结果 |
| `zh-CN` | 中文 | 中文 | 写入 `"language": "zh-CN"` |
| `en` | English | English | 写入 `"language": "en"` |

主题选项为：

| 持久化值 | 中文界面标签 | English UI label | 行为 |
| --- | --- | --- | --- |
| 属性缺失 | 跟随系统 | Follow system | 删除 `settings.json` 的 `theme`，当前页按 `prefers-color-scheme` 解析 |
| `light` | 浅色 | Light | 写入 `"theme": "light"` |
| `dark` | 深色 | Dark | 写入 `"theme": "dark"` |

两个选项均复用页面已有的自定义下拉组件，不使用操作系统原生 `<select>` 弹层。语言切换后当前页立即重新渲染；主题切换后立即应用且不改变布局。后续 UI 和新启动的 CLI 使用新语言设置，已运行的其它 CLI 进程不动态切换。`<html lang>` 随解析结果更新为 `zh-CN` 或 `en`。语言和主题互相独立。

浏览器不独立读取 `navigator.language`，也不在 `localStorage` 保存语言或主题偏好；页面以服务端返回的全局设置为唯一持久化真值。服务端在返回首页时把 `theme` 偏好写入 `<html>` 数据属性，样式表加载前的本地脚本据此选择显式主题或系统主题，避免刷新闪烁。

UI 不直接读写文件，由服务端提供：

| 方法与路径 | 入参 | 成功输出 | 错误 |
| --- | --- | --- | --- |
| `GET /api/settings` | 无 | `{ "language": null\|"en"\|"zh-CN", "resolvedLanguage": "en"\|"zh-CN", "theme": null\|"light"\|"dark" }` | 读取/解析失败返回 500 + 固定英文技术诊断 |
| `PATCH /api/settings` | 至少包含 `language` / `theme` 之一；值为各自合法字符串或 `null` | 同 `GET`；`null` 删除对应属性 | 空对象、未知字段、非法值返回 400；读写失败返回 500 |

`PATCH` 请求体必须是对象，可一次修改一个或两个字段；未出现的字段保持不变。未知字段拒绝为 400，防止调用方误以为其它设置已经生效。`language` 的 `null` 表示跟随环境，`theme` 的 `null` 表示跟随系统；`resolvedLanguage` 表示当前实际语言。服务端不返回 `resolvedTheme`，因为系统主题由浏览器媒体查询决定。这是 UI 配置基础设施，不要求增加对等 CLI 子命令。

页面中的导航、按钮、表头、字段、弹窗、空态、成功/警告、前端校验和服务端领域错误均必须进入中英文字典。网络/技术失败可显示本地化概要 + 固定英文技术详情。

## 共享领域错误

store、workflow、run、launch 等共享层不得将中文或英文领域句子当作错误身份。需要本地化的错误必须携带稳定错误码、结构化参数，以及可选的固定英文技术详情/原始下游错误。

CLI 根据统一语言解析结果渲染错误码 + 参数；HTTP API 返回同一错误码 + 参数，由 UI 渲染。不得依赖 `err.Error()` 的翻译文本做 `errors.Is`、HTTP 状态码映射或程序分支。

HTTP 错误统一使用以下机器稳定 envelope；字段名、错误码和参数名不本地化：

```json
{
  "error": {
    "code": "workflow_validation_failed",
    "problems": [
      {
        "path": "definition.nodes[1].id",
        "code": "reserved_node_id",
        "params": { "id": "START" }
      }
    ]
  }
}
```

`code` 必填；无参数时省略 `params`，不是批量校验错误时省略 `problems`，无技术详情时省略 `technicalDetail`。每个 problem 的 `path`、`code` 与 `params` 都是机器稳定值。领域错误不在响应里附带已翻译句子。技术失败使用稳定技术错误码，`technicalDetail` 保存 conduct 固定英文诊断或原样下游文本；UI 可据 `code` 显示本地化概要，但不得翻译 `technicalDetail`。

| 错误码 | 参数 | 中文 | 英文 |
| --- | --- | --- | --- |
| `workflow_not_found` | `name` | `工作流 <name> 不存在` | `workflow <name> does not exist` |
| `workflow_already_exists` | `name` | `工作流 <name> 已存在` | `workflow <name> already exists` |
| `working_directory_not_found` | `path` | `工作目录不存在：<path>` | `working directory does not exist: <path>` |

## run-summary.md

- 运行开始时把已解析的 `en` / `zh-CN` 记入该 run，后续 resume 与 summary 沿用，不在收尾时重新读取全局设置。
- `run.json.language` 是必填字段，只允许 `en` / `zh-CN`。缺失或非法表示运行记录损坏，读取、续跑和重写时均以固定英文技术诊断失败；不按旧版中文、当前设置或环境变量补默认值。
- summary 生成后不因设置、环境变量或 UI 语言变更而重写；UI 切换只改页面外壳。
- 用户需求、节点名、原始错误和 AI 产物始终原样保留。

## 测试要求

- 设置解析覆盖：文件缺失、属性缺失、语言/主题全部合法值、无法读取、损坏 JSON、非对象、非字符串与不支持值。
- 优先级覆盖 `settings.language` > `LC_ALL` > `LC_MESSAGES` > `LANG` > `en`。
- 设置写入覆盖语言与主题全部选项、部分更新、未知属性保留、原子替换和失败时原文件不变。
- CLI 在 `en` 下覆盖成功、空态、用法/领域错误、校验失败和技术诊断；排除用户/引擎原文后，conduct 产生的产品文案不得含汉字。
- CLI 在 `zh-CN` 下验证产品文案为中文，同时技术诊断仍为英文。
- UI 中英文字典 key 集合必须一致；设置页使用自定义下拉；切换后遍历列表、编辑器、运行详情、弹窗和错误态，并核对 `<html lang>` 与 `<html data-theme>`。
- 每个共享领域错误码都覆盖 CLI/UI 中英文渲染。
- `run-summary.md` 在两种语言下验证模板，并确认运行期间切换全局语言不影响该 run。
- 缺失或非法 `run.json.language` 必须按损坏记录处理，不得生成 summary、恢复运行或写回兼容默认值。
