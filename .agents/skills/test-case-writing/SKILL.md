# 测试用例写作

测试用例文档描述**怎么手工验证一个功能点是否正确**——照着文档一步步敲命令，对照预期结果就能判断通过 / 失败，不需要额外背景知识。它是给人或 agent 照做的可执行清单，**不是** `go test` 单元测试（那是代码），也不是功能说明书。一篇文档覆盖一个命令 / 功能域，内含若干独立用例。

## 硬性约定（MUST）

1. **隔离执行、无副作用**：每个用例在独立临时目录里跑（`WORK=$(mktemp -d)`），结束在〈清理〉里 `rm -rf "$WORK"`。禁止污染仓库、全局配置（`~/.config`、`~/.conduct` 等）、家目录或任何共享状态。只读用例（如查版本）可不建临时目录，但同样不得写任何文件。
2. **可重复、幂等**：固定输入、固定版本，同一用例连跑两遍结果必须一致。不依赖上一个用例留下的状态、不依赖网络 / 时钟 / 随机值；若功能天然涉及这些，在〈前置〉里固定住（造 fixture、传定值）。
3. **预期具体可判定**：每个用例写明**退出码**和**关键输出 / 产物**（stdout / stderr 片段、生成的文件及内容）。预期不能是「应该正常」这种没法对照的话。
4. **归一化非确定输出**：版本号、临时路径、时间戳、耗时、并发顺序等每次都变的内容，不能逐字写进预期。用占位符（`conduct <VERSION>`）、只匹配子串、或先排序 / 过滤再对照，并注明忽略哪些字段。见〈处理非确定输出〉。
5. **傻瓜式步骤**：步骤是编号的、可直接复制粘贴的命令，第一步做什么、第二步做什么写清楚；不留「配置好环境」这类需要读者自行脑补的空档。
6. **文件名英文、不记时间、诚实标注**：文档存 `docs/test-cases/<name>.md`，小写 + 连字符（如 `cli-run.md`）；正文不写编写 / 修改时间（交给 git）；未实现 / 未达标的功能，〈预期〉写 spec 规定的**目标行为**，另加一句**〈现状注〉**标注其当前真实报错与退出码——既不假装能跑通，功能落地后又只需删注、无需改预期。
7. **自足、可独立完成（无外部人工依赖）**：整条用例必须由**单个执行者**（一名刚入职、不懂业务的程序员，或一个 AI）凭文档 + 本机**独立**跑完，中途不依赖任何**第三方带外操作**——不得出现「让管理员去后台点『审批通过』」「等同事合并 PR」「找运维开通账号」「等对方回邮件 / 短信验证码」这类必须另一个人介入才能继续的步骤。判据：把这条用例单独发给一个零上下文的人或 AI，他手上只有这份文档和一台机器，能不能从头跑到尾？不能就是不合格。若被测功能天然含这种关卡，在〈前置〉里把它**预置或 stub 掉**（提前造好「已审批」状态的数据、备好凭据 / token、用假服务顶替外部系统、用固定测试账号），让执行者全程自己掌控。实在无法自足的功能，不适合写成手工用例——注明原因，交给能覆盖它的那一层（带真实环境的集成 / 端到端测试）。

**别造「只能人工」的用例来逃避自动化**：执行者默认可能是 AI，「需要人在真实终端敲」「人工目视确认」这类说法**不是**把步骤标成手工项的理由。缺的几乎总是**机制或工具**、不是「人」——`stdin` 要真终端就用伪终端接管（`pty.openpty()` + `communicate(timeout=)` 超时守卫，超时即判挂起 FAIL）；要浏览器交互就用浏览器自动化工具（如 playwright MCP）；缺某个命令就**明确开口要**，而不是 SKIP。判据从「人能不能跑」升级为「一个只有文档 + 一台机器 + 可申领工具的 AI，能不能把它跑成无人值守自动化」——**能却写成手工，就是失职**。真正只能靠人的极少（肉眼判断视觉像素、插拔物理外设之类），那种才按上一段注明原因、交给对应测试层。
8. **用户视角、黑盒操作（前置状态不伪造）**：用例站在**使用者**角度，只经系统对外暴露的入口（CLI / API / UI）操作与断言，**不碰内部实现**。构造前置状态必须走**真实的对外操作**，**严禁直接伪造与内部实现耦合的数据**——不往数据库 INSERT、不手写系统私有的落盘文件（存储 JSON / 缓存 / 状态文件）来「摆拍」出某个状态。例：测「购物车下单」，先经页面 / 接口把商品**真的加入购物车**再下单；而不是往 MySQL 插一行伪造「购物车有货」再下单——伪造既跳过了真实链路（加购这步本身的 bug 就漏测），又与内部 schema 死耦合（实现一改、用例即废）。**边界与优先级**：系统**自身**的内部状态绝不伪造；对它依赖的**外部系统**（第三方接口、外部引擎 / 服务），**优先用真实的**（哪怕慢些、花些成本），仅当真实调用确实不可行（烧钱不可控 / 输出不确定 / 环境不可得）才退用测试替身，且优先选真实依赖的沙箱 / 测试账号而非纯假造。

## 功能覆盖率（动笔前先规划，MUST）

用例的价值不在「跑通一条命令」，而在**覆盖功能的行为空间**。动笔前先把被测功能的行为列成清单，逐项映射到用例——只写 happy path 是最常见的失职。清单至少含四类：

1. **正常路径**：核心功能按预期工作。
2. **边界与错误路径**：对着 spec 的校验规则表**逐条**列——每个必填字段、每条命名 / 取值约束、每组互斥关系、每个越界值，都要有一个专门触发它失败的用例。别只挑显眼的一两个，校验规则是最容易「测了个别、漏了一片」的地方。
3. **数据流转（不只看退出码）**：当功能是「把 A 变成 B」「把前一步的输出喂给后一步」这类转换时，必须断言**数据真的流动了**——构造一个可辨识的输入，在下游 / 输出里验证它按预期出现，而不是只看退出码为 0。例：验一个「引用上一步产物」的功能，要用**至少两步**、让第二步引用第一步的输出，断言第一步的输出确实出现在第二步的输入里；只跑单步、只看命令成功，是验不出这个引用的。
4. **特性叠加（组合里挑关键路径）**：两个特性同时作用时的顺序、优先级、相互影响，最容易藏 bug 也最容易漏测。为关键组合各写一个用例，断言其**完整可观察结果**，而非只数数量。例：两种循环 / 重试机制叠加在一起时，要断言最终跑出来的**完整执行顺序**；只对「总共跑了几步」打卡，会放过「顺序错位但步数恰好不变」的 bug。

**别用退化夹具测关系型功能**：功能的本质若是「多个元素之间的关系」（元素间的引用、循环、依赖顺序），夹具就必须含足够多的元素去触发那个关系。只放一个元素就验不出元素间的引用、只跑一轮就验不出循环累积——退化夹具会制造「测过了」的假象。

**分层但不遗漏**：手工用例这层重在验对外可观察的行为（退出码 / 输出 / 产物文件）；深藏在内部的算法精确正确性（如某种展开 / 求值的逐步结果）往往更适合交给单元测试。若某行为在手工层难以稳定断言，记下来交给单测覆盖，并确认它**确有某一层在覆盖**——不要因为「手工不好测」就整个漏掉。

## 环境准备（每篇跑一次）

```bash

```

## 单个用例结构（五段）

每个功能点写成一个用例，标题 `TC-<编号> <一句话>`，固定五段：

- **目的**：验证哪个功能点、哪条预期行为。
- **前置**：建隔离环境、造 fixture、设定值。只读用例可写「无」。
- **步骤**：编号命令，可复制粘贴。
- **预期**：退出码 + 关键输出 / 产物；非确定内容按〈处理非确定输出〉归一化。
- **清理**：删临时目录 / 恢复现场。只读用例可写「无」。

示例（`version` 已实现，直接对照；全局 `--version` 标志尚未接线，用〈现状注〉标注其当前真实行为——见 MUST 第 6 条约定）：

```markdown
### TC-001 version 打印版本号

- **目的**：验证 `conduct version` 输出版本号且退出码为 0。
- **前置**：无（只读）。
- **步骤**：
  1. `"$CONDUCT" version`
- **预期**：
  - 退出码 `0`。
  - stdout 形如 `conduct <VERSION>`（`<VERSION>` 为 SemVer 或 `dev`，不逐字比对）。
- **清理**：无。

### TC-002 --version 全局标志打印版本号

- **目的**：验证全局 `--version` 打印版本号并退出 0（spec 全局约定）。
- **前置**：无（只读）。
- **步骤**：
  1. `"$CONDUCT" --version; echo "exit=$?"`
- **预期**：
  - 退出码 `0`；stdout 与 TC-001 同格式（`conduct <VERSION>`）。
  - **现状注**：该标志尚未接线，当前实际返回 `unknown flag: --version`、退出 `2`。上为 spec 目标行为，待落地后本用例即通过。
- **清理**：无。
```

## 处理非确定输出

预期里凡是每次运行都会变的内容，必须归一化，否则用例无法重复对照：

- **版本号 / 构建信息**：写占位符 `<VERSION>`，只校验格式或前缀，注明不逐字比对。
- **临时路径**：只匹配文件名或相对片段，别写死 `/var/folders/...` 这种 mktemp 路径。
- **时间戳 / 耗时**：从对照中排除，或用 `sed`/`grep -v` 过滤后再比。
- **无序输出**：先 `sort` 再对照，别依赖行序。
- **多行大段输出**：只断言必须出现的关键行（`grep -q`），不要求逐字整块相等。

## 完稿自检

- **可复制**：把每个用例的步骤命令顺序粘进终端就能跑，无缺失变量、无需手工补步骤。
- **独立可完成、无手工逃避**：通篇没有需要别人代劳的步骤——外部审批 / 协作 / 开通类依赖都已在〈前置〉里预置或 stub；也没有拿「需人在终端敲 / 人工目视」当借口逃避自动化的步骤——真终端用 `pty`、浏览器用自动化工具、缺命令就开口要（见 MUST 第 7 条）。逐条自问「一个只有文档 + 一台机器 + 可申领工具的 AI，能不能把它跑成无人值守自动化？」
- **用户视角**：前置状态都靠真实对外操作（命令 / 接口 / UI）建立，没有直接往 DB 插数据、手写系统私有存储文件来摆拍；断言只看对外可见的输出 / 产物，不读内部实现细节（见 MUST 第 8 条）。
- **真跑一遍**：实际执行用例，确认退出码、输出、产物与〈预期〉一致；不一致就改预期或改结论，不硬凑。
- **无残留**：用例跑完 `$WORK` 已删除，仓库 `git status` 干净，全局配置未被改。
- **幂等**：同一用例连跑两遍结果一致。
- **诚实**：未实现 / 有已知缺陷的功能，〈预期〉写 spec 目标、另以**〈现状注〉**标注其真实现状（当前报错 / 退出码，见 MUST 第 6 条）；不把设计当已完成。
- **对齐 spec**：若有对应 spec，用例覆盖其列出的主要行为与错误路径，无悬空引用。
- **覆盖率复盘**：对着〈功能覆盖率〉的行为清单逐项打勾——每条校验规则、每个数据流转、每个关键特性组合，是否都有用例或已明确交给单测？把缺的列出来，别默认「跑通了就是覆盖了」。

## 沉淀新规范

新确立的**通用**测试约定 → 追加到本 SKILL.md；某类命令专属的**长夹具或大段样例输出** → 放 `references/` 单独成文，从这里一行引用。
