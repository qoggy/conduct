# Spec 写作

spec 说明一个 CLI / API / 功能**对外长什么样**——有哪些命令、接口或行为，各自的**入参与出参**、行为与约束。它是设计与评审用的文档，**不**规定实现代码怎么写，也不是用户手册。写成自包含、与代码同步的仓库内文档：读者只读这一篇就能理解并据以实现。

## 硬性约定（MUST）

1. **自包含引用**：只引用本仓库内的文件（相对仓库根的路径，如 `internal/cli/run.go`）和网络链接（URL）。禁止引用其它仓库、其它工作空间、本机绝对路径下的文档——它们对 clone 本仓库的人不存在；需要的背景直接写进 spec。
2. **文件名用英文**：spec 存 `docs/specs/<name>.md`，文件名小写 + 连字符（如 `cli-commands.md`），不用中文文件名。
3. **不记录编写时间**：不写「写于 YYYY-MM-DD」之类 dateline，正文不记录本文档的编写 / 修改时间（示例数据里的时间戳不算）。时间线交给 git。
4. **诚实标注状态**：严格区分 已实现 / 未实现 / 待确认，不把设计写得像已完成；用〈实现状态〉说明与代码的差距。
5. **Markdown**：章节标题不加数字编号；不放目录。

## 骨架

按需增删，核心是把每条命令 / 接口的入参出参写全：

- **一句话说明**：这是什么、面向谁。
- **设计前提**：可推翻的地基假设，前置；已拍板的作为事实陈述。
- **总览表**：所有命令 / 接口一览。
- **逐条详述**：每条写全——用途、用法、**入参**（位置参数、选项，各标 必填 / 默认 / 取值）、**出参**（正常输出、错误、退出码 / 状态码、副作用）、示例。
- **数据 / schema**（若涉及）：字段、类型、必填、默认、约束。
- **退出码 / 状态码约定**（若涉及）：统一一张表。
- **实现状态**：诚实对齐代码现状。
- **待确认**：开放项交用户拍板，别硬编一个假决定。

## 出入参写全

这是 spec 的核心价值。每条命令 / 接口的两侧都要写全：入参标 必填 / 默认 / 取值；出参含 正常输出、错误、退出码 / 状态码、副作用。凡在「用法」里列出的选项，都必须在下文被定义——列了不解释即为缺口。

## 完稿自检

- **交叉引用**：每个「见〈X〉」都对应唯一章节，无悬空锚点。
- **删除无残留**：删掉的命令 / 选项全文不留旧引用（总览表、示例、待确认、退出码表）。
- **表 ↔ 正文一致**：退出码表覆盖正文用到的每种码；总览与逐条集合一致。
- **术语统一**：同一事物全文一个叫法；符合〈硬性约定〉（无外部引用、英文文件名、无编写时间、状态诚实）。

## 沉淀新规范

新确立的**通用**约定 → 追加到本 SKILL.md；某类 spec 专属的**长模板或大段示例** → 放 `references/` 单独成文，从这里一行引用。
