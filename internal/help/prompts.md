# 写好节点 promptTemplate

conduct 把工作流拆成节点、逐个交给 AI 引擎执行。每个节点的 `promptTemplate` 渲染后作为该节点的输入送进引擎。这份文档讲怎么把它写好。

## 一、节点看得到什么（决定你能引用什么）

节点跑在**互相隔离**的独立会话里——一个节点看不到别的节点的对话历史和工具调用过程。它能看到的只有：

1. 渲染后的 `promptTemplate`（变量见第二节）
2. 它用工具（如读文件）从工作目录磁盘上读到的内容
3. 内循环重跑时，conduct 自动追加的上一轮 evaluator 反馈段

关键：`{{<节点id>}}` 拿到的是被引用节点的**最终输出文本**（引擎跑完返回的收尾产物），不是它的完整对话记录。

### evaluator 节点（节点内循环）

给节点配 `evaluator` 会触发「写 → 评 → 改」的内循环，按 `loopCount` 固定重跑相应轮数（不是由评测结论决定是否继续）。evaluator 运行时，conduct **自动**在它输入末尾追加被评节点本轮的产物：

```
## Artifact under review

<artifact_under_review>
……被评节点的产物……
</artifact_under_review>
```

所以 **evaluator 的 `promptTemplate` 不需要自己写 `{{<被评节点id>}}`**——产物已经拼在末尾了。

### 被内循环重跑的节点

节点进入下一轮重跑时，conduct **自动**在它输入末尾追加上一轮的评测反馈：

```
## Previous evaluator feedback

<previous_evaluator_feedback>
……上一轮 evaluator 的反馈……
</previous_evaluator_feedback>
```

所以**重跑节点的 `promptTemplate` 也不用自己写引用**去取反馈。注意：每次重跑都是**全新会话**，看不到上一轮的工具调用过程，只有这段反馈文本。

## 二、模板变量

渲染是**纯字符串替换**，不做转义。三类变量：

- `{{sys.userPrompt}}` → 用户需求（运行时必然注入）
- `{{sys.cwd}}` → 工作目录（运行时必然注入）
- `{{<节点id>}}` → 该 id 节点的产物

`{{<节点id>}}` 在这些情况替换成**空字符串**：节点还没运行、运行失败、产物为空、或内循环首轮自己引用自己。

同一节点被多轮运行时，`{{<节点id>}}` 取的是**最近一次成功**的产物；更早轮次的产出取不回来（要追溯历史轮次，查该次 run 的 trace）。

### 输出字面量 `{{...}}` → 用 `\{{...}}` 转义

若 prompt 确需让 AI 输出 `{{xxx}}` 这个字面量（比如教它写 Jinja / Handlebars 模板、生成含双花括号的配置），写成 `\{{xxx}}`——渲染时去掉前导反斜杠、原样输出，不当变量替换。已知 / 未知 key 都适用。

## 三、最佳实践

### promptTemplate 是写给 AI 的，不是写给人的

它渲染后**整段送进 LLM**——每个字都占 token、占上下文、分散注意力。所以 `promptTemplate` 里只写给 AI 的指令；写给人看的备注、设计取舍、「为什么这么写」的说明，不要塞进来。

### 动态内容用 XML 标签包裹

`{{sys.userPrompt}}` 和 `{{<节点id>}}` 塞进来的内容几乎都是 markdown——里面的 `## 标题` 会和你 `promptTemplate` 自己的 `## 标题` 同级混在一起，模型分不清哪些是给它的指令、哪些是待处理的数据。用 XML 标签（非 markdown 的边界符号）包起来，模型就能清楚「标签内是数据、标签外是指令」：

```
<user_prompt>
{{sys.userPrompt}}
</user_prompt>

<plan>
{{plan}}
</plan>
```

标签名按数据语义自取。（conduct 自动追加的 evaluator 反馈段、待评产物段，本身就是这么包的。）

**特例**：若某个变量注入的是**短而结构化的输出**（如一个 0–100 的分数、单个词、一个状态标记），它不含 markdown 标题、不会和指令混淆，XML 包裹可省（当然包上也无妨）。

### 同一变量别重复插入

变量是字面量替换，写几次就复制几份。重复插入的真正代价不是 token 钱，而是**挤占有限上下文、稀释模型注意力**——注入内容越大越明显。默认每个动态变量**只出现一次**；下游要再用，让 AI 从前文的 XML 段里回看即可。确需插两次，应是权衡过的有意决定。

同样地，上面那种**短而结构化的输出**（如一个分数）即便被下游重复插入，成本也可忽略，不必拘泥这条。
