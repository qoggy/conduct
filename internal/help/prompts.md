# 写好节点 promptTemplate

conduct 把工作流拆成节点 + 边构成的 DAG（有向无环图），逐个节点交给 AI 引擎执行。每个节点的 `promptTemplate` 渲染后作为该节点的输入送进引擎。这份文档讲怎么把它写好。

## 一、节点看得到什么（决定你能引用什么）

节点跑在**互相隔离**的独立会话里——一个节点看不到别的节点的对话历史和工具调用过程。它能看到的只有：

1. 渲染后的 `promptTemplate`（变量见第二节）
2. 它用工具（如读文件）从工作目录磁盘上读到的内容

关键：`{{<节点id>}}` 拿到的是被引用节点的**最终输出文本**（引擎跑完返回的收尾产物），不是它的完整对话记录。

每个 agent 节点在一趟运行里（含中断后 `run resume` 续跑）**至多执行一次**。要表达「写 → 评 → 改」这类精修，把每一步拆成独立节点用边串起来：`写(a) → 评审并修正(b)`，让 `b` 读 `{{a}}` 一次性产出改进版。

## 二、模板变量

渲染是**纯字符串替换**，不做转义。支持以下变量：

- `{{sys.userPrompt}}` → 用户需求（运行时必然注入）
- `{{sys.cwd}}` → 工作目录（运行时必然注入）
- `{{sys.runId}}` → 本次运行的 run id（运行时必然注入）
- `{{<节点id>}}` → 该 id 节点的产物

`{{<节点id>}}` 的 `<节点id>` 必须是**本节点的上游祖先 agent 节点**（沿边可达的前驱）——这在入库校验时就强制检查，引用不存在的节点、非祖先节点、或标记节点 `{{START}}` / `{{END}}`（二者无产物）一律被拒绝、不允许落盘。因此一份合法定义里，节点跑到时其模板引用的祖先必已成功产出（调度器保证：节点就绪即其全部祖先都已完成）。

**并行分支的产物不自动汇聚**——`START` 扇出的多个节点各自独立产出，互不可见彼此的产物；要收口就让下游节点在其 `promptTemplate` 里逐个引用每条分支的 `{{id}}`（如 `c` 同时读 `{{a}}` 与 `{{b}}`）。

### 输出字面量 `{{...}}` → 用 `\{{...}}` 转义

若 prompt 确需让 AI 输出 `{{xxx}}` 这个字面量（比如教它写 Jinja / Handlebars 模板、生成含双花括号的配置），写成 `\{{xxx}}`——渲染时去掉前导反斜杠、原样输出，不当变量替换。已知 / 未知 key 都适用。

## 三、并行分支避免写盘冲突

conduct 的调度是「就绪即跑」：`START` 同时扇出到多个节点、或某节点的多个后继依赖同时满足时，它们会在同一时刻并行开跑。并行开跑的节点共享这次运行的同一个 `cwd`：如果两个并行节点都要读写磁盘上同一份工作树（改同一批文件、跑同一个构建产物目录），谁先谁后落盘不确定，可能互相覆盖或冲突。工作树隔离要在设计工作流 / 写提示词时主动做，有两种方式：

1. **把并行分支设计成互不冲突的任务**：例如让 `START` 扇出的几个节点都是只读评审（多个角度各写一份 review，互不改动代码）、或按文件 / 目录边界拆分好、彼此不触碰对方文件的独立任务（如「前端」和「后端」各自的目录）。
2. **在提示词里让各分支的 AI 各自开一个独立工作区**：显式指示该节点先 `git worktree add` 出一个只属于自己的目录再动手，完成后把结果交给下游节点合并回主工作树。例如：

   ```
   在开始修改前，先执行：
   git worktree add ../wt-<分支标识> -b <分支名>
   之后所有改动都在 ../wt-<分支标识> 里进行，不要碰当前工作目录。
   ```

   下游收口节点再读各分支的产物（各自的 diff / 变更摘要）合并到主工作树。

就绪即跑是既定行为；并行分支的写盘冲突面由用户在提示词侧控制。

## 四、最佳实践

### promptTemplate 是写给 AI 的，不是写给人的

它渲染后**整段送进 LLM**——每个字都占 token、占上下文、分散注意力。所以 `promptTemplate` 里只写给 AI 的指令；写给人看的备注、设计取舍、「为什么这么写」的说明，不要塞进来。

### 动态内容用 XML 标签包裹

`{{sys.userPrompt}}` 和 `{{<节点id>}}` 塞进来的内容几乎都是 markdown——里面的 `## 标题` 会和你 `promptTemplate` 自己的 `## 标题` 同级混在一起，模型分不清哪些是给它的指令、哪些是待处理的数据。用 XML 标签（非 markdown 的边界符号）包起来，模型就能清楚「标签内是数据、标签外是指令」：

```
<user_prompt>
{{sys.userPrompt}}
</user_prompt>

<plan>
{{plan}}
</plan>
```

标签名按数据语义自取。

**特例**：若某个变量注入的是**短而结构化的输出**（如一个 0–100 的分数、单个词、一个状态标记），它不含 markdown 标题、不会和指令混淆，XML 包裹可省（当然包上也无妨）。

### 同一变量别重复插入

变量是字面量替换，写几次就复制几份。重复插入的真正代价不是 token 钱，而是**挤占有限上下文、稀释模型注意力**——注入内容越大越明显。默认每个动态变量**只出现一次**；下游要再用，让 AI 从前文的 XML 段里回看即可。确需插两次，应是权衡过的有意决定。

同样地，上面那种**短而结构化的输出**（如一个分数）即便被下游重复插入，成本也可忽略，不必拘泥这条。
